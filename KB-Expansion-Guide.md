# Knowledge-Base Expansion Guide
## 从综述到顶会：知识库驱动的研究方法论

> **适用阶段**：Stage 1（文献检索）+ Stage 2（Interim Survey 写作）并行推进
> **长期目标**：从研究小白出发，建立扎实的领域知识基础，最终发表顶会论文
> **核心原则**：知识库是"外部研究记忆"——每一条笔记都必须服务于可检验的研究问题

---

## 零、研究路径总览

### 0.1 阶段划分与完成标准

```
Stage 0  文献地图构建（已完成）
   ├── 输出：对三篇 Survey 的系统性笔记与交叉分析
   └── 完成标志：能用自己的话描述领域的关键争议与已有回答

Stage 1  深度文献阅读（当前阶段）
   ├── 输出：P0/P1 论文的完整精读笔记 + 更新后的 comparison-matrix + gap-tracker
   └── 完成标志：见 §7.1

Stage 2  Interim Survey 写作（与 Stage 1 并行）
   ├── 输出：10-15 页综述 + 结构化 Evidence Map
   └── 完成标志：见 §7.2

Stage 3  选题确立与方法论设计
   ├── 输出：可检验的 Research Question + 方法论草案
   └── 完成标志：见 §7.3

Stage 4  实验设计与原型实现
Stage 5  论文写作与投稿
```

### 0.2 从 Interim 到顶会：能力迁移路径

Interim Survey 不只是课程交付物——它是你顶会论文的**Related Work 和 Motivation 章节的草稿**。两者的核心差异在于：

| 维度 | Interim Survey | 顶会论文 |
|------|--------------|---------|
| Research Question | 方向性描述 | 精确的、可检验的假设 |
| Gap 证据 | 合理推断 | 原文引用 + 实验数字支撑 |
| 方法论 | 草案设计 | 消融实验验证的设计选择 |
| Taxonomy | 分类框架 | 经 ME/CE 验证的、有新颖性的分类 |

**Interim 的核心任务**：把 Stage 0 的"Survey 理解"升级为"有原文支撑的论断"，为 Stage 3 的方法论设计奠基。

## 一、Research Question 的形式化（Stage 1 前置任务）

> **为什么先做 RQ？** RQ 决定你需要读哪些论文、需要提取什么信息、需要证明什么。没有 RQ 的文献阅读是漫无目的的信息收集。

### 1.1 RQ 层级结构

一个研究项目的 RQ 分三层：

```
Main RQ（总研究问题）
   ├── Sub-RQ 1（文献综述层面：现有工作是什么？）
   ├── Sub-RQ 2（Gap 层面：现有工作缺什么？）
   └── Sub-RQ 3（方法论层面：如何填补这个 Gap？）
```

**当前草稿 RQ**（来自 taxonomy-draft.md）：

> "LLM-based GUI Agent 如何从经验中学习，以在不同任务、应用和用户之间积累可复用的操作知识？"

这是一个**研究方向**，尚不是严格意义的 RQ。形式化需要补充：

| 要素 | 当前状态 | 需要补充 |
|------|---------|---------|
| **范围边界** | "GUI Agent" 宽泛 | 哪类 GUI？哪类任务？哪类平台？ |
| **可检验性** | 描述性问题 | 能被什么实验回答？ |
| **现有工作对比** | 未说明 | 为什么现有工作无法回答这个 RQ？ |
| **预期产出** | 未明确 | 理论框架？系统原型？性能评估？ |

### 1.2 RQ 质量检验清单

完成 Stage 1 后，用以下标准检验你的 RQ 是否达到顶会水平：

- [ ] **Specific**：RQ 能指定一类方法和一类评估场景
- [ ] **Answerable**：存在可行的实验设计来回答它
- [ ] **Novel**：现有论文没有直接回答它（需要 Gap 证据支撑）
- [ ] **Feasible**：在你的资源条件下可在 1-2 个 benchmark 上验证
- [ ] **Impactful**：回答它能推进领域认知，而不只是填一个技术细节

> **实践建议**：在 Stage 1 开始时写下"Provisional RQ"，随着精读论文不断修订，Stage 2 结束时应能输出 Final RQ。

## 二、文献检索与范围定义

### 2.1 检索协议

**检索数据库**：Semantic Scholar（主）、arXiv、Google Scholar、ACL Anthology（NLP 方向）、ACM DL

**时间范围**：2022年至今（LLM-based GUI Agent 的实质性起点）；Background 章节可追溯至 2019 年

**核心关键词组合**：

| 领域交叉 | 检索字符串示例 |
|---------|-------------|
| GUI Agent + Memory | `"GUI agent" memory`, `"web agent" episodic`, `"computer use" experience accumulation` |
| GUI Agent + Self-Evolving | `"GUI agent" self-improvement`, `"web agent" continual learning`, `"GUI agent" skill learning` |
| 三者交叉 | `"GUI agent" memory self-improvement`, `"web agent" procedural knowledge` |
| 关键系统名 | `AppAgent`, `MobileGPT`, `AutoDroid`, `AWM`, `SkillWeaver`, `Friday` |

**搜索饱和标准**：当连续检索 20 篇新论文中，有 ≥ 15 篇已在知识库或已知要排除，停止该关键词组合的检索。

### 2.2 Inclusion / Exclusion 标准

这是保证综述 claim 可辩护的基础。每篇加入知识库的论文，必须满足以下标准：

**Inclusion（全部满足才纳入）：**
- [ ] 论文的主要研究对象是 LLM-based Agent（而非传统 RL Agent 或规则系统）
- [ ] 与以下至少一个主题直接相关：GUI 操作 / 任务自动化 / Agent Memory / Agent 自我进化
- [ ] 有可复现的实验结果（有 benchmark 数字或明确的用户研究）
- [ ] 发表在同行评审会议/期刊，或 arXiv 预印本（有后续引用或来自知名团队）

**Exclusion（满足任意一条则排除）：**
- [ ] 仅针对单一封闭系统（如特定 App 的 RPA 脚本），无泛化意图
- [ ] 没有 LLM/MLLM 作为核心组件
- [ ] 纯综述性质且不提供新方法/新框架（三篇 seed survey 除外）

> **记录要求**：每篇被排除的论文，在 gap-tracker.md 的"范围外论文"列表中记录排除原因。这在写综述时可以防御审稿人质疑。

### 2.3 引用追踪策略

**Backward tracing**（从已读论文的参考文献出发）：
- 优先追踪 P0 论文引用的、被标记为"直接基础"的论文
- 特别关注：方法论上你计划借鉴的论文（AWM、SkillWeaver、Generative Agents、MemGPT）

**Forward tracing**（谁引用了你已读的论文）：
- 在 Semantic Scholar 中查看 citation count > 10 且近 6 个月发表的引用者
- 这是发现最新 SOTA 的最快路径

**关键作者追踪**：当某个团队在你的领域连续发表 ≥ 2 篇高质量论文，追踪该作者/团队的完整发表列表。

## 三、论文精读协议

> **目标**：不只是"读懂"一篇论文，而是提取出你写综述和设计方法论所需的精确信息。

### 3.1 三遍阅读法

```
第一遍（15-20 分钟）：定位阅读
   读：Title → Abstract → Introduction 末段 → Conclusion → Figure 1
   记：填写笔记模板的 [元信息] + [一句话总结] + [Provisional 优先级]
   判：这篇论文与我的 RQ 的关系（直接相关 / 间接相关 / 仅背景）

第二遍（1-2 小时）：结构化阅读
   读：全文主体，跳过数学证明，但读懂每个模块的"设计选择"
   重点提取：
     ① 作者解决的问题的精确表述（通常在 Introduction § 3-4 段）
     ② 方法的关键设计决策：为什么这样设计而不是另一种？
     ③ Baseline 的选择：为什么选这些 baseline？有没有漏掉重要的？
     ④ 实验结果的局限：作者自述了什么？我观察到什么？
   记：完整填写笔记模板

第三遍（按需，仅 P0 论文）：实验深度阅读
   读：Appendix、实验设置细节、消融实验
   记：[可复用的实验设计] + [方法论细节]
   思：这个实验设计能用于验证我的方法吗？需要怎样修改？
```

### 3.2 精读的核心追问（针对顶会准备）

对每篇 P0/P1 论文，必须能回答以下问题：

**关于问题设定：**
- 这篇论文假设的"前提条件"是什么？我的研究场景是否满足这些前提？
- 如果去掉他们最关键的假设，他们的方法还成立吗？

**关于方法设计：**
- 这个方法的 design choice 有多少个？哪些是核心的，哪些是辅助的？
- 消融实验验证了哪些 design choice？哪些没有被验证？（未验证的 = 潜在的 Gap）

**关于实验：**
- 他们的 benchmark 能衡量他们声称的能力吗？有没有系统性偏差？
- 如果换一个 benchmark，结论会变吗？

**关于关系到我的研究：**
- 这篇论文最强的证据（原文 + 数字），能不能支持我的某个 claim？
- 这篇论文的 limitation，是不是我要解决的问题？

### 3.3 阅读饱和判断

Stage 1 的文献阅读何时结束？满足以下条件时：

- [ ] P0 论文（含 seed papers 的待读清单）已全部完成第二遍或第三遍
- [ ] 连续精读 3 篇新论文，未出现新的 Gap 类型（理论饱和）
- [ ] 每个已识别的 Gap（A 类）都有 ≥ 2 篇独立论文的原文证据支撑
- [ ] comparison-matrix.md 已更新，覆盖 ≥ 15 个代表性系统

## 五、Taxonomy 构建协议

> Taxonomy 是 Survey 的核心学术贡献之一。一个好的分类框架不只是"把论文分组"，而是揭示领域的内在结构。

### 5.1 Taxonomy 的质量标准

**ME（Mutual Exclusivity，互斥性）**：每篇论文只能归入一个类别，不能同时属于两个。如果发现论文"跨类"，说明分类维度设计有问题。

**CE（Collective Exhaustiveness，完备性）**：分类体系能容纳领域内所有已知论文，以及未来可能出现的论文，不会有"无法分类"的情况。

**Justification（维度选择的论证）**：每个分类维度必须能回答"为什么是这个维度而不是另一个"。维度的选择应该：
- 反映领域的核心争议（而非任意切割）
- 与 Research Question 直接相关（维度揭示 Gap 的位置）
- 有文献支撑（维度的定义来自领域内的已有概念，或有清晰的操作化定义）

### 5.2 当前 Taxonomy 的审查（What × When × How）

现有三维框架（来自 taxonomy-draft.md）：

| 维度 | 定义 | ME 检验 | CE 检验 | 选择理由 |
|------|------|---------|---------|---------|
| **What**（学什么）| Skills / World / Episodes / User | 待验证 | 待验证 | 对应 AM 的认知类型 |
| **When**（何时学）| 任务前/中/后/跨任务 | 待验证 | 待验证 | 对应 SE 的时序框架 |
| **How**（如何复用）| 精确/相似/泛化/注入 | 待验证 | 待验证 | 检索机制的分类 |

**验证方法**：将知识库中所有已记录的 GUI Agent 系统（≥ 15 个）填入分类矩阵，检查：
- 是否有系统无法被分类（CE 失败信号）
- 是否有系统同时属于两个格子（ME 失败信号）
- 矩阵中空白的格子是否真的是"逻辑上不可能存在"，还是"尚未有人做"（Gap 信号）

### 5.3 Taxonomy 与 Gap 的关系

一个好的 Taxonomy 的空白格子 = 潜在的 Research Gap。但不是所有空白都有价值：

```
有价值的空白：
   ├── 逻辑上可行（技术上没有根本障碍）
   ├── 领域内有需求（有论文明确指出此方向的重要性）
   └── 尚无人填补（知识库检索确认）

无价值的空白：
   ├── 技术上不可行（存在根本性障碍）
   └── 领域内无需求（没有论文讨论此问题）
```

## 六、Gap 的系统性识别与证据分级

### 6.1 Gap 证据分级标准（操作化定义）

| 等级 | 定义 | 操作化标准 | 在综述中的表述方式 |
|------|------|---------|----------------|
| **强（A）** | 多个独立来源的直接证据 | ≥ 2 篇论文原文明确指出此 Gap，或 ≥ 3 篇论文的实验数据间接证明 | "As [A] and [B] explicitly acknowledge..." |
| **中（B）** | 单一来源或间接证据 | 1 篇论文原文提及，或从对比矩阵中观察到系统性缺失 | "While [A] hints at..., no work has directly..." |
| **弱（C）** | 合理推断 | 仅基于逻辑推理，无原文支撑 | 不应在综述中直接声明，需要先找到证据升级 |

> **规则**：所有进入 Interim Survey 的 Gap claim 必须达到 B 级或以上。进入顶会论文的 Motivation 必须达到 A 级。

### 6.2 Gap 条目标准格式

```markdown
### Gap [ID]: [简短标题]

**陈述**（一句话，精确描述缺失的是什么）

**证据等级**: A / B / C

**正向证据**（支持此 Gap 存在的）:
- [论文 A, Section X, p.Y]: "[原文引用]" → 说明 ___
- [论文 B] 的实验数据：在 ___ benchmark 上，___ 类系统的成功率仅为 ___，
  而 ___ 类系统达到 ___，差距说明 ___

**反向证据检验**（是否有论文已经部分解决此 Gap？）:
- [论文 C] 尝试了 ___，但局限于 ___（仍未完全解决 Gap 的原因）
- 结论：[论文 C] 的工作不构成对此 Gap 的有效填补，原因是 ___

**可行性评估**:
- 技术可行性：高 / 中 / 低（理由：相关技术在 ___ 场景已验证）
- 所需资源：___ 数据集 + ___ 基础框架
- 与 RQ 的关联：直接对应 Sub-RQ ___

**优先级**: P0 / P1 / P2
```

### 6.3 反证检验的重要性

在顶会 review 中，最常见的拒稿理由之一是"此 Gap 已被 [某论文] 解决"。防御这个攻击的方法是：

1. 主动搜索最接近你 Gap 的现有论文
2. 精确说明它们"解决了什么"和"没有解决什么"
3. 在综述中主动承认这些相关工作，并清晰划定你的 Gap 与它们的界限

## 七、论证链：从 Evidence 到 Claim

### 7.1 CER 结构（Claim - Evidence - Reasoning）

综述中的每一个重要论断都应该遵循 CER 结构：

```
Claim（你的论断）
   ├── Evidence（支撑这个论断的原文证据）
   │   ├── 引用 1: "[原文]"（论文 A, Section X, p.Y）
   │   └── 引用 2: 实验数字（论文 B, Table X）
   └── Reasoning（为什么这些证据支持你的 Claim）
       └── 证据说明 ___, 因此可以推断 ___
```

**一个具体示例**：

> **Claim**: 当前 GUI Agent 缺乏跨任务的 Procedural Memory 机制
>
> **Evidence**:
> - AppAgent (Yang et al., 2024, §3.2): "The explored knowledge is stored as a static JSON file and is not updated during task execution."
> - 对比矩阵统计：23 个 GUI Agent 系统中，21 个（91%）无持久化 Procedural Memory
>
> **Reasoning**: 静态存储意味着成功的操作序列无法被系统地归纳为可复用的技能；91% 的覆盖率证明这不是个例而是系统性缺失。

### 7.2 Evidence Map 模板

每个综述章节的核心论断，在写作前建立证据映射：

```markdown
## [章节名] Evidence Map

| Claim | Evidence（论文 + 位置 + 原文摘录） | Reasoning | 证据强度 |
|-------|--------------------------------|---------|---------|
| [论断 1] | [A §X p.Y]: "..." | 因为... | A / B / C |
| [论断 2] | [B Table 3] + [C §4.2 p.8]: "..." | ... | A / B / C |
```

**写作规则**：如果某个 Claim 的证据强度为 C，不要写进综述，先回到知识库找更强的证据。

## 八、阶段完成标准

### 8.1 Stage 1 完成标准（可以进入 Stage 3 前必须满足）

## 九、检索与阅读的实操执行顺序

```
Week 1-2: 建立精读流水线
  ├── 确立 Provisional RQ，写入 taxonomy-draft.md
  ├── 完成 Inclusion/Exclusion 标准定义
  ├── 筛选所有候选论文，分配 P0/P1/P2
  └── P0 论文开始第一遍阅读

Week 3-4: P0 论文精读
  ├── AWM、SkillWeaver（A-1 Gap 核心技术来源）
  ├── Generative Agents、MemGPT（A-2 Gap 蓝图）
  ├── Friday（唯一接近 A-3 的 GUI 系统）
  └── 持续更新 comparison-matrix + gap-tracker

Week 5-6: Taxonomy 验证 + Gap 证据化
  ├── 将所有已读论文填入 Taxonomy 矩阵，检验 ME/CE
  ├── 为每个 A 类 Gap 完成反证检验
  ├── 建立 Evidence Map（对应 Interim 章节）
  └── 升级 Provisional RQ

Week 7-8: Interim Survey 写作
  ├── 先写 Related Work（最有文献支撑的部分）
  ├── 再写 Introduction（需要确定你在 Methodology 章节要说什么）
  ├── 最后写 Methodology（此时 RQ 和 Gap 应已稳定）
  └── 全文用 Evidence Map 核查每个 Claim 的证据

Week 9+: Stage 3 准备
  ├── RQ Final 版本
  ├── 方法论草案
  └── 实验设计框架
```
