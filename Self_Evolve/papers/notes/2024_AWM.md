## One-line Summary
Wang et al. 提出 Agent Workflow Memory（AWM），把历史轨迹抽象成可复用 workflow 并作为记忆注入 agent，在 WebArena 和 Mind2Web 上分别实现 51.1% 与 24.6% 的相对提升，并在跨任务/跨站点/跨域设置下持续优于基线。

## Problem Setting
- **Core problem**: 作者指出现有 agent “fail to grasp ... extract and learn reusable task workflows” 并且逐任务独立求解，难以长期适应（Section 1, p.1）。
- **Assumptions**:
  - agent 具备可执行的基础动作集合（如 `CLICK`/`TYPE`）与文本记忆（Section 2.1, p.2）。
  - 任务以自然语言指令 + 多步交互形式给出，且可观测状态并迭代执行动作（Section 2.1, p.2）。
  - online 模式下可用评估器判断历史预测是否正确，以筛出可用于归纳 workflow 的经验（Section 2.3, p.3-4）。
- **Insufficiency of existing approaches**: “they do not learn from past successes and failures, and are therefore unable to adapt over time” (Section 1, p.1)。

## Core Method
- **Method overview**:

AWM 的核心是把“经验轨迹”提升为“可复用流程记忆”。每个 workflow 包含两部分：一个高层目标描述 `d`（描述该流程解决什么子任务），以及一组步骤轨迹 `P^d`（每步含环境描述、推理、动作）。为了可迁移性，workflow 会把实例值抽象成变量（如 `{product-name}`），而不是保留原样（Section 2.2-2.3, p.3）。

在执行层面，AWM 把归纳得到的 workflows 注入 agent memory，再用于后续决策。作者给了两种运行范式：`offline`（先用训练/标注样本一次性归纳 workflows，再测试）与 `online`（测试流式过程中持续 induce -> integrate -> apply）。online 形态能在无额外标注样本时持续扩展记忆，形成滚雪球式学习（Section 2.3, p.3-4）。

实现上，作者主推 LM-based induction：从多个经验中抽取公共子流程并生成抽象 workflow；还做了 rule-based 对照。实验证明 workflow 作为“抽象子程序”比“直接检索完整历史轨迹”更稳健，尤其在跨分布泛化下优势更明显（Section 3-4, p.5-10）。

- **Key Design Choices**:

| Design Decision | Author's Choice | Author's Rationale | Ablation Verified? |
|----------------|----------------|--------------------|--------------------|
| 记忆单元 | 存 workflow（目标描述 + 步骤）而非原始完整轨迹 | 降低 example-specific 偏置，提升跨任务复用 | Yes（vs Synapse / rule-based，Table 3,6） |
| Workflow 表达 | 抽象变量化（如 `{location}`）+ 子流程粒度 | 提升跨实例可迁移性 | Yes（rule vs LM，Table 5/6） |
| 运行范式 | 同时支持 offline 与 online | offline 利用高质量样本；online 适配无标注/分布漂移场景 | Yes（Table 4, Table 11） |
| 引导方式 | 先作为 memory 注入；另测作为高层 action | memory 注入更稳，action 扩展收益有限 | Yes（Table 9） |
| 状态表征 | workflow 中默认使用 NL state 描述 | 比仅用 HTML 更有效，且上下文更紧凑 | Yes（Table 8） |
| 数据组织 | 以 website 为单位构建 workflows | 贴合 web agent 跨站点分布差异 | Partial（实验设置说明，无专门敏感性分析） |

- **Core difference from prior work**: 相比直接检索历史样例（如 Synapse）或手工写 workflow（SteP），AWM 重点在“自动归纳 + 抽象子流程 + 持续更新记忆”，并在无人工 workflow 的前提下取得更好效果。

## Key Results

| Benchmark | This Method | Strongest Baseline | Δ | Notes |
|-----------|------------|-------------------|---|-------|
| WebArena (Total SR) | AWM: 35.5 | BrowserGym: 23.5 | +12.0 abs / +51.1% rel | 同时优于带人工 workflow 的 SteP 33.0（Table 1, p.5） |
| WebArena (#Steps) | AWM: 5.9 | BrowserGym ax-tree: 7.9 | -2.0 steps | 成功率提升同时步骤更短（Table 1, p.5） |
| WebArena Cross-template (Total SR) | AWM: 33.2 | SteP: 32.1 | +1.1 abs | 在非重叠模板子集仍领先（Table 2, p.6） |
| Mind2Web Cross-task (StepSR, GPT-4) | AWM offline: 45.1 | MindAct: 36.2 | +8.9 abs | ElemAcc 也提升（50.6 vs 41.6）（Table 3, p.7） |
| Mind2Web Cross-website (StepSR, GPT-4) | AWM online: 33.9 | MindAct: 30.1 | +3.8 abs | online 略优于 offline（33.9 vs 33.7）（Table 4, p.8） |
| Mind2Web Cross-domain (StepSR, GPT-4) | AWM online: 35.5 | MindAct: 18.6 | +16.9 abs | 分布差越大，online 优势越明显（Table 4, p.8） |

- **Key ablation findings**:
  - LM-based workflow induction 相比 rule-based 在 Mind2Web 上更优（StepSR 45.1 vs 43.4；SR 4.8 vs 2.0），说明抽象子流程的质量关键（Table 6, p.9）。
  - Workflow 记忆中的状态信息以 NL 描述优于 HTML；混合 NL+HTML 反而下降（StepSR 34.6 -> 32.9），说明上下文噪声与长度是实质问题（Table 8, p.9）。
  - 将 workflow 扩展为 action space 仅小幅增加 StepSR（文中写 +1.3），总体 SR 提升有限，且仅 18.5% 任务调用 workflow actions（Section 5, p.10）。
  - **未充分 ablate**：workflow 检索/选择策略细节、memory 容量上限、online evaluator 误判敏感性。

- **Failure cases**:
  - 动态环境中的弹窗选项是显式失败模式：预定义 workflow action 在“看不到中间状态变化”时不够灵活（Figure 7, Section 5, p.10）。
  - offline 与 online workflow 直接混合并非叠加收益，存在兼容性问题（Table 11, p.16）。

## Limitations
- **Author-stated limitations**: 作者指出 workflow action 在动态界面上“is not flexible enough”，并建议 future work 引入 real-time state access 或 dynamic execution loops（Section 5, p.10）。
- **My observed limitations**:
> ⚠️ NEEDS YOUR INPUT: 初步观察如下，请结合你的研究目标确认。
> 1. 评测集中在 text-based web benchmarks（WebArena/Mind2Web），对视觉 GUI、移动端、多模态操作的外推证据不足。  
> 2. online 依赖 evaluator 判断历史轨迹是否“正确”，但文中缺少 evaluator 误差传播分析。  
> 3. offline+online 融合出现负向干扰（Table 11），说明“记忆合并策略”仍是开放问题。  
> 4. 任务级 SR 在 Mind2Web 上整体仍偏低（如 4.8），离稳定工业部署还有距离。
- **Experimental design gaps**:
  - 缺少计算开销/延迟分析（workflow induction 额外成本未系统报告）。
  - 缺少更强闭源 agent baseline 的统一重跑对比（部分对照来自已发表方法设置）。
  - 没有评估 memory 增长到更大规模后的上下文拥塞与长期稳定性。
