# Modular Architecture Evolution

> Paper Section IV-C (pages 9)

---

## Core Concept

Modular Architecture Evolution focuses on optimizing the **structural modules** that mediate agent-environment interactions — not the content of interactions (Static Knowledge) or the experience gained (Dynamic Experience), but the *mechanisms* themselves.

> "This paradigm centers on optimizing the structural modules that mediate agent-environment interactions — enhancing both interaction efficiency and adaptability."

Three categories:

```
Modular Architecture Evolution
├── 1. Interaction Protocol Evolution   (how agent manages context/history)
├── 2. Memory Architecture Evolution    (how agent organizes stored experience)
└── 3. Tool-Augmented Evolution         (how agent creates/uses tools)
```

---

## 1. Interaction Protocol Evolution

### The Core Problem

Long-horizon tasks generate massive interaction histories that overflow context windows. Naive approaches (concatenate everything) fail due to:
- Token limits
- Diminishing returns from growing context
- Irrelevant information degrading reasoning quality

### Cognitive Abstraction Protocols

These methods distill interaction history into compact, useful representations:

| System | Key Innovation |
|---|---|
| **Think-in-Memory** [Liu et al., 2023] | Stores inductive thoughts to avoid contradictions across reasoning steps |
| **Memory-of-Thought (MoT)** [Li & Qiu, 2023] | Filters high-confidence reasoning paths for reuse |
| **ReadAgent** [Lee et al., 2024] | Inspired by fuzzy trace theory — compresses text into "Gist Memory" (key ideas, not verbatim) |

### Dynamic Management Protocols (OS-inspired)

OS-level virtual memory principles applied to agent context:

| System | Key Innovation |
|---|---|
| **MemGPT** [Packer et al., 2023] | Virtual memory paging — moves data between "main memory" (context) and "external storage"; LLM acts as OS kernel |
| **MemoryBank** [Zhong et al., 2024] | Incorporates "Ebbinghaus Forgetting Curve" — recent memories are stronger, old ones decay |
| **AgentFold** [Ye et al., 2025] | Proactive context folding — folds irrelevant trajectories to prevent overflow |
| **LightMem** [Fang et al., 2025] | Lightweight and efficient memory-augmented generation |
| **General Agentic Memory via Deep Research** [Yan et al., 2025] | Just-in-time memory reconstruction using deep research process |

### Key Insight: Memory as Active Reconstruction

> "Rather than passively accumulating raw interaction history, this paradigm emphasizes distilling trajectories into concise summaries and salient cues, regulating what information should be retained or exposed at different stages of reasoning."

## 2. Memory Architecture Evolution

### From Static Database to Adaptive System

Traditional memory = static key-value store. Evolved memory = dynamic, self-organizing network that:
- Decides what to store based on usefulness
- Adapts its own organizational structure
- Handles conflicting or redundant information

### Key Systems

| System | Key Innovation |
|---|---|
| **A-MEM** [Xu et al., 2025] | Agentic memory treating memory as dynamic network inspired by "Zettelkasten" note-taking method |
| **Mem0** [Chhikara et al., 2025] | Mimics human cognition to intelligently decide on data retention; production-grade |
| **MemEvolve** [Zhang et al.] | Meta-evolution: agents autonomously adapt routing strategies and memory logic based on task feedback |
| **Memory as Action** [Zhang et al., 2025] | Autonomous context curation — memory management as a learnable decision process |
| **Mem-α** [Wang et al., 2025] | Learning memory construction via reinforcement learning |
| **CAM** [Li et al., 2025] | Constructivist approach: real-time flexible assimilation of cognitive schemas |

### Memory as Reinforcement Learning Decision

An emerging direction: frame memory operations as RL actions:

```
State: Current working memory + query
Action: {store, retrieve, update, compress, forget}
Reward: Task performance improvement
Policy: Learned optimal memory management strategy
```

Systems like MemEvolve and Memory-as-Action take this approach, moving from heuristic toward principled memory control.

## 3. Tool-Augmented Evolution

### The Progression: Use → Create → Manage

```
Level 1: Tool Use           ReAct, PAL (use predefined APIs)
Level 2: Tool Optimization  WebGPT (learn when to use which tool)
Level 3: Tool Creation      LATM, CREATOR (generate new tools)
Level 4: Tool Library Mgmt  VOYAGER (persistent reusable skill library)
```

### Level 1-2: Reasoning-Execution Synergy

| System | Key Innovation |
|---|---|
| **ReAct** [Yao et al., 2022] | Interleaves reasoning (Thought) with acting (Action) — foundation pattern |
| **WebGPT** [Nakano et al., 2021] | Browser-assisted QA with human feedback |
| **PAL** [Gao et al., 2023] | Offloads complex logic to external Python interpreters |

### Level 3: Autonomous Tool Creation

| System | Key Innovation |
|---|---|
| **LATM** [Cai et al., 2023] | Generates tools for lightweight agents from demonstrations |
| **CREATOR** [Qian et al., 2023] | Separates abstract design from execution — builds specialized toolsets |
| **CRAFT** [Yuan et al., 2023] | Customizes LLMs by creating and retrieving specialized toolsets |
| **TOOLMAKER** [Wölflein et al., 2025] | Applies autonomous tool creation to scientific workflows with automated debugging |

### Level 4: Lifelong Skill Library

**VOYAGER** [Wang et al., 2023]:
- Stores **successful actions as executable code** in a skill library
- Grows indefinitely through lifelong learning in Minecraft
- Enables 15.3× faster progression vs baselines

**Alita** [Qiu et al., 2025]:
- Scalable evolution through minimally predefined architecture
- Dynamically generates tools from compact core via standardized context-based interaction
- Enables open-ended capability expansion

**Process**:
```
Task → Execute → Success?
  ↓ Yes                ↓ No
Abstract skill    Debug/modify
into library      strategy
    ↓
Future similar task retrieves skill
```

## Summary: Three Sub-paradigm Comparison

| Sub-paradigm | What Evolves | Goal | Key Challenge |
|---|---|---|---|
| Interaction Protocol | How history is managed | Efficient context use | Info loss vs overflow |
| Memory Architecture | How experience is organized | Optimal retention/retrieval | Deciding what to keep |
| Tool-Augmented | What tools the agent can use | Expanding action space | Tool quality/reliability |
