# Memory Subjects — Who Memory Serves

> Paper Section 3.3 (pages 17–20)

Memory subjects define **whose information** the memory system is designed to capture and support. This dimension is orthogonal to both substrate and cognitive mechanism — the same external vector store can serve user-centric or agent-centric purposes.

---

## Two Subjects

| Subject | Memory Stores | Primary Goal |
|---|---|---|
| **User-Centric** | User preferences, identity, interaction history, dialogue context | Personalize agent behavior to individual users across sessions |
| **Agent-Centric** | Task trajectories, outcomes, learned skills, domain knowledge | Improve agent task performance and capabilities over time |

---

## 3.3.1 User-Centric Memory

> Memory designed to capture and support **the user's information** — their preferences, history, and context.

### Core Use Cases

#### Long-Term Personalization
- Maintain consistent user profile across sessions and time
- Capture stable preferences (communication style, domain expertise, dietary restrictions, etc.)
- Example: A-Mem builds dynamic user memory that evolves with each interaction

#### Privacy-Preserving Memory
- Protect stored user data through access control and selective sharing
- Challenge: balancing utility of long-term memory with privacy requirements
- Critical in healthcare, legal, and financial domains

#### Memory Management in Dialogues
- Keep user-relevant context consistent across long conversation sessions
- Prevent early-context forgetting (user's name, preferences, stated goals)
- Example: RoleLLM maintains persona-consistent memory; MemoCRS tracks user preferences for recommendations

### Key Challenges for User-Centric Memory

1. **Preference drift**: Users change their minds over time — memory must adapt, not freeze
2. **Credit assignment**: Which past interactions are relevant to the current request?
3. **Memory staleness**: When should old preferences be overwritten vs. preserved?
4. **Privacy and security**: Persistent user memory → attack surface (memory poisoning, unauthorized access)
5. **Conflicting signals**: User says one thing, does another — which to store?

### Representative Works

| System | Focus |
|---|---|
| RoleLLM (Wang et al., 2024e) | Persona consistency through static prompt-encoded memory |
| MAUMB (Hou et al., 2024) | Multi-user, multi-agent memory management |
| MemoCRS (Xi et al., 2024) | Conversational recommendation with user preference memory |
| RET-LLM (Modarressi et al., 2023) | Explicit read/write memory for long-term dialogue |
| Zep (Rasmussen et al., 2025) | Production-grade persistent memory for AI assistants |
| A-Mem (Xu et al., 2025e) | Dynamic, evolving user memory |
| Echo (Liu et al., 2025i) | User-centric memory for personalized response |
| EpiCache (Kim et al., 2025b) | Efficient episodic caching for user interactions |

## 3.3.2 Agent-Centric Memory

> Memory designed to capture and support **the agent's own information** — its task histories, learned skills, and accumulated expertise.

### Core Use Cases

#### Long-Horizon Tasks
- Store key intermediate states to support extended multi-step execution
- Avoid re-doing completed work; resume from where the agent left off
- Example: JARVIS-1 maintains multimodal memory of completed sub-tasks in Minecraft

#### Domain-Specific Long-Tail Solutions
- Real-world problems exhibit long-tail phenomena — rare but important patterns
- Agent stores customized one-off solutions and reapplies when similar cases arise
- Examples:
  - Software debugging: specific configuration issues, dependency conflicts
  - Scientific research: specialized experimental setups for sub-fields
  - Information search: narrowly scoped context-dependent problems

#### Cross-Task Knowledge Transfer
- Accumulate durable knowledge across tasks and episodes
- Abstract task-agnostic knowledge → generalize to new domains
- Key distinction: not replaying specific trajectories, but distilling into **transferable abstractions**
- Example: Agent KB (Tang et al., 2025b) — cross-domain experience framework aggregating high-level strategies

#### Strategy and Skill Learning
- Retain environment-grounded procedural memories encoding **how to efficiently execute** multi-step actions
- Web agents: reuse successful multi-step browsing policies
- GUI agents: store interface-specific action sequences (menu traversal, widget manipulation, error recovery)
- Embodied agents: executable skills + control policies respecting physical dynamics
- Example: Voyager (Wang et al., 2025c) skill library; JARVIS-1; AWM

### Key Challenges for Agent-Centric Memory

1. **What to store**: Not all trajectories are worth retaining — when does an experience generalize?
2. **Skill abstraction**: How to extract transferable knowledge from specific task executions
3. **Forgetting good information**: As memory grows, how to prune without losing important edge-case solutions
4. **Cross-domain transfer**: How to recognize when a stored strategy applies to a new domain

### Representative Works

| System | Focus |
|---|---|
| JARVIS-1 (Wang et al., 2024q) | Multimodal memory for embodied agent in Minecraft |
| Buffer of Thoughts / BoT (Yang et al., 2024b) | Procedural thought templates stored for reuse |
| AWM (Wang et al., 2025u) | Workflow memory for web automation |
| HippoRAG 2 (Gutiérrez et al., 2025) | Human-inspired episodic indexing for agent memory |
| Cognee (Markovic et al., 2025) | Knowledge graph memory for agents |
| Branch-and-Browse (He et al., 2025b) | Tree-structured action memory for web exploration |
| GridMM (Wang et al., 2023d) | Spatial memory for grid-world navigation |

## User-Centric vs. Agent-Centric: Key Differences

| Dimension | User-Centric | Agent-Centric |
|---|---|---|
| **Primary beneficiary** | The human user | The agent system |
| **What's stored** | User preferences, identity, dialogue history | Task trajectories, skills, domain knowledge |
| **Temporal scope** | Across user sessions (weeks/months/years) | Across task instances and episodes |
| **Success metric** | User satisfaction, personalization quality | Task completion rate, efficiency |
| **Privacy concern** | High (PII, sensitive preferences) | Lower (task-level data) |
| **Update trigger** | User interaction and feedback | Task outcomes and agent reflection |

## Both Subjects in the Same System

Many real-world systems maintain **both** simultaneously:

```
Foundation Agent
    ├── User-Centric Memory
    │   ├── User profile (name, preferences, history)
    │   ├── Dialogue context (past conversation key points)
    │   └── Sensitive user data (with access control)
    │
    └── Agent-Centric Memory
        ├── Task trajectories (what was tried, what worked)
        ├── Skill library (reusable procedures)
        └── Domain knowledge (accumulated expertise)
```

Example: LD-Agent (Li et al., 2025a) — maintains both long-term user interaction memory and agent-side task knowledge for dialogue systems.
