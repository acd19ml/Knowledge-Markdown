# Memory Substrates — Where Memory Lives

> Paper Section 3.1 (pages 8–12)

Memory substrates are the **storage mechanisms** that retain historical knowledge produced during agent–human interaction. Two broad categories: External and Internal.

---

## External Memory (§3.1.1)

> "External memory refers to any memory substrate that stores knowledge, information, and past experience **outside** the agent model's parameters or state. The agent can explicitly read from and write to external memory via retrieval and update operations."

### Key Properties

- Operates independently of neural network weights (gradient updates don't affect it)
- Clear separation: **computation** (in LLM) vs. **knowledge** (in external DB)
- Scalable and flexible: can be added, replaced, or inserted without retraining
- Preserves past information in original form → reduces hallucination, avoids knowledge cutoff

### Drawbacks

- Retrieval adds **latency** (especially in multi-turn iterations)
- Retrieval can be **unreliable** (noisy, irrelevant results inject noise)
- Storage and indexing costs **escalate** as memory grows
- Requires explicit memory management: summarization, selective retention, forgetting, deduplication, pruning

### Four Implementation Types

#### 1. Vector Index

**How it works**: Embed memory items into a shared vector space → approximate nearest-neighbor search → top-k items appended to prompt.

**Core insight**: Memory as a **geometric problem** — each item is a point in continuous space; queries find closest semantic neighbors.

**Indexing techniques**: HNSW (hierarchical navigable small-world), IVF (inverted file), PQ (product quantization)

**Dominant paradigm**: RAG (Retrieval-Augmented Generation) framework

**Representative**: Lewis et al. (2020), RAG framework; most current agent memory systems

#### 2. Text-Record

**How it works**: Memory as explicit **text documents** — running "core summary" + episodic logs, periodically summarized, selectively copied into prompt.

**Structure**:
- **Semantic list**: discrete facts, modifiable via insert/update/delete
- **Episodic list**: chronological ledger of timestamped events
- **Core summary**: persistent, high-level summary

**Advantages**: Transparent, fast integration, human-readable
**Disadvantages**: Requires meticulous summarization and pruning

**Representative**: Generative Agents (Park et al., 2023), SCM, MemGPT

#### 3. Structural Store

**How it works**: Memory in **explicit schemas** — retrieved via symbolic queries or traversal.

Three sub-types:

| Type | Storage | Retrieval | Use Case |
|---|---|---|---|
| **Relational tables** | SQL rows | SQL joins, indexes | Facts, preferences, events |
| **Graph-based memory** | Nodes (entities) + Edges (relations) | Semantic embeddings + keyword search + traversal | Knowledge relationships, event connections |
| **Tree-based memory** | Hierarchical nodes with summaries + embeddings | Multi-level abstraction | Long conversation tasks, hierarchical knowledge |

**Knowledge graph specifics**: Nodes = entities, Edges = relationships. On new info: detect edge changes, resolve conflicts, maintain valid graph structure (Jiang et al., 2025c).

**Tree specifics**: Each node stores aggregated summary + embedding; deeper branches = more specific detail. Dynamic mechanism supports multi-level abstraction + efficient retrieval (Rezazadeh et al., 2025b, MemTree).

**Representative**: ChatDB (Hu et al., 2023), HippoRAG 2, Cognee, MemTree, PISA

#### 4. Hierarchical Store

**How it works**: Multiple dedicated **memory modules**, each with own schema and strategy, coordinated by a meta-memory manager.

**Example modules**:
- Core memory: persistent persona + user facts
- Episodic memory: time-series, sensitive events
- Semantic memory: abstract concepts
- Procedural memory: step-by-step instructions (JSON sequences)
- Resource memory: documents, media
- Knowledge store: sensitive/private info (with strict access control)

**Retrieval methods per module**: embedding similarity, BM25 match, string match

**Meta-memory manager**: directs queries to correct store, aggregates results

**Advantages**: Modularity, separation of concerns, scalability, specialized schemas
**Disadvantages**: Architectural complexity, potential latency, data consistency challenges

**Representative**: MIRIX (Wang & Chen, 2025), MemGPT, MAUMB, multi-component systems

## Internal Memory (§3.1.2)

> "Internal memory refers to information stored directly within the model's architecture, encompassing both the persistent knowledge embedded in its parameters (parametric memory) and the working states utilized during inference."

### Three Implementation Types

#### 1. Weights (Parametric Memory)

**How it works**: Knowledge/experience embedded directly into neural network parameters through training or editing. Agents recall information without external retrieval.

**Three update strategies**:

| Strategy | Method | Key Idea |
|---|---|---|
| **Continual learning** | Incremental weight updates | Regularization / soft masks to avoid catastrophic forgetting |
| **Model editing** | Targeted parameter editing | Locate decisive mid-layer weights → rank-one updates (ROME, MEMIT) |
| **Distillation** | Compress context into weights | Train student to replicate prompted teacher behavior; "in-context distillation" |

**Advantages**: High-speed access, tight integration with reasoning, no retrieval overhead
**Disadvantages**: Expensive to update; catastrophic forgetting risk; interference when editing

**Representative**: MemoryLLM (Wang et al., 2024m), SELF-PARAM, Titans, LMLM, MAC, WISE

#### 2. Latent-State Memory

**How it works**: Carry forward and reuse **intermediate hidden states** across steps/segments. Earlier activations directly impact later computation even when raw tokens are no longer in the window.

**Key property**: Created during runtime activations → used → discarded unless exported. Supports **within-session coherence** but doesn't persist across sessions.

**Research directions**:
- Reconstructing hidden representations across layers to reinforce earlier context
- Caching compact subsets of hidden-state vectors across segments
- Generating memory trigger + memory weaver tokens (synthetic latent tokens fed back in)
- Structured modulation of hidden-state transitions to maintain semantic alignment
- Compressive memory integrated into attention mechanism (Munkhdalai et al., 2024)
- Fast-weight updates during inference (Zhu et al., 2025b)

**Resource cost**: Must hold both parameters AND intermediate states; scales with model depth, batch size, precision.

**Representative**: Titans (Behrouz et al., 2024), HMT, MemGen, RET-LLM

#### 3. KV Cache

**How it works**: Transformer inference-time memory. Cache per-layer attention keys and values from previous tokens; reuse for subsequent tokens instead of recomputing.

**Without KV cache**: Recompute K/V matrices for all previous tokens at every step → O(n²) compute for long sequences.

**With KV cache**: Only compute for freshly generated token → significant speedup for long outputs.

**Cost**: Higher memory usage; linear scaling with sequence length.

**KV Cache compression research**:
| Approach | Key Idea |
|---|---|
| Dynamic eviction | Identify "heavy hitter" tokens contributing most to attention; evict the rest |
| Pattern-based clustering | Identify attention patterns; cluster important features |
| Per-layer importance ranking | Different layers need different cache sizes; binary search under global budget |

**Representative**: vLLMPA (Kwon et al., 2023), ChunkKV (Liu et al., 2025j), StreamingLLM, EpiCache

## Tradeoffs and Design Guidance (§3.1.3)

| Property | Internal (Weights) | Internal (Latent/KV) | External |
|---|---|---|---|
| Access speed | Fast | Very fast | Slower (retrieval) |
| Capacity | Fixed | Limited (linear) | Scalable |
| Update cost | Very high (retraining) | Ephemeral (reset) | Low (insert/delete) |
| Cross-session persistence | Yes | No | Yes |
| Noise risk | Low | Low | High (grows with scale) |
| Best for | Stable general knowledge | Within-session state | Long-term experience |

### Hybrid Architectures (Current Best Practice)

- **Parametric memory**: inherent facts and world knowledge
- **Latent memory**: fast short-term reasoning within a session
- **External memory**: scalable storage of experiences across sessions

This reflects a broader shift from **static persistent knowledge** to **dynamic adaptive** context management.

---

## Research Gaps and Extensions

- **Noise management**: As external memory grows, retrieval quality degrades — active area for pruning/summarization methods
- **Hybrid coordination**: How do parametric + external memories interact during inference? How to decide what goes where?
- **Memory efficiency**: KV cache compression remains an active research area (ChunkKV, EpiCache)
- **Continual learning + agents**: Combining parametric weight updates with external memory management without catastrophic forgetting
