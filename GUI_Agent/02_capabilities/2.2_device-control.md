# 3.1.2 设备控制（Device Control）

> 论文 Section 3.1.2（Page 15）

设备控制是 GUI Agent 将决策转化为实际操作的执行层能力。

---

## 两类控制方式

```
设备控制
   │
   ├── Code-Based（代码生成）
   │     生成 Python/API 调用脚本 → 执行器运行
   │
   └── UI-Based（界面模拟）
         模拟人类点击/滑动/输入 → 直接操作 UI 元素
```

---

## Code-Based Agent

**核心思路**：让 LLM 生成代码（Python 脚本或 API 调用），由执行器运行。

| 系统 | 代码生成方式 | 说明 |
|------|------------|------|
| **WebAgent** | HTML-T5 提取片段 → FLAN-U-PaLM 生成 Python | 操作网站 |
| **WebGPT** | 微调 GPT-3 直接调用 Microsoft Bing Search API | 搜索增强问答 |

**优势**：
- 操作精确、可重复
- 适合有完整 API 的应用（网页、标准应用）

**局限性**：
- 并非所有应用的 API 可访问（封闭应用）
- 不同应用 API 差异大，通用性差

---

## UI-Based Agent

**核心思路**：模拟人类的鼠标点击、键盘输入、滑动等操作，直接与 GUI 元素交互。

- 需要预定义**动作空间**（Action Space）：点击、输入文字、滑动、返回等
- 以自然语言或坐标形式选择动作
- **绝大多数 GUI Agent 采用此方式**（兼容性最强）

**动作空间示例**：
```
Click(element)
Type(text)
Scroll(direction, amount)
Navigate(url)
Press(key)
Wait()
Done()
```

---

## 选择依据

| 场景 | 推荐方式 |
|------|---------|
| 网页应用（DOM 可访问） | Code-Based（效率高） |
| 移动端封闭 App | UI-Based（唯一选择） |
| 桌面通用任务 | UI-Based（通用性强） |
| 需要精确 API 操作 | Code-Based |

---

## Personal Research Notes

- [ ] UI-Based 方案在**跨平台迁移**时代价极低（无需了解 API），是 Self-Evolving 场景的更优选择
- [ ] Code-Based 生成代码的**正确性验证**是一大挑战——与 Section 6.2 的 Feasibility 问题直接相关
- [ ] 目前两类方式均缺乏**操作历史记忆**：同样的操作失败后，Agent 下次依然可能重蹈覆辙
